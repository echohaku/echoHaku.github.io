Git 系列分享 - p.6 正式提交一個commit囉

October 22, 2025
#Git

在Git 系列分享 - p.5 檔案追蹤並納入紀錄！（git status、git add）中，我們學會了如何使用 git add 將變更從工作目錄移動到暫存區（Staging Area），完成變更的打包。

現在，我們將進行工作流程中的最後一步：正式提交（Commit），為這次變更留下一個永久且不可磨滅的歷史快照。

git commit：為變更留下永久快照

git commit 是 Git 流程中最關鍵的動作。它會將暫存區的快照永久寫入你的 Git 倉庫 (Repository)，形成專案歷史上的一個新版本節點。

什麼是一個 Commit？

  • 定義： 一個 Commit 就是一個「有意義的、邏輯完整的工作單元」。
  • 劃分原則： 我們應該以「目的」來劃分 Commit，而不是以「時間」。
    • 重點強調： Commit 的大小並不重要！它可以是一個微小的 Bug 修正（小版本），也可以是一個完整的核心功能開發（大版本）。關鍵在於：這個 Commit 必須只包含單一目的的變更。
    • 好的 Commit： 專注於單一目標。例如：「修正使用者登入頁面的 CSS 樣式」或「新增資料庫連線功能」。
    • 不好的 Commit： 包含了不相關的變更。例如：「修正 Bug + 調整排版 + 實驗性新功能」。
  • 重要性： 當你確定暫存區的變更已經完成了一個獨立的任務時，才應該執行 Commit。

Commit Message 的重要性

Commit Message 就像是這個版本變動的「摘要與目的」，它是版本歷史中最重要的註解。

  • 給未來的自己看： 當幾個月後，你需要追溯某個功能或 Bug 的來源時，清晰的訊息能讓你快速定位。
  • 給團隊成員看： 隊友能透過訊息快速了解你做了什麼，避免重複工作或產生衝突。

💡 優秀的 Commit Message 寫法建議：

  1. 首行（標題）： 簡潔扼要地描述本次提交的目的（通常不超過 50 個字符）。
  2. 使用動詞開頭： 例如：feat: (新增功能), fix: (修正錯誤), docs: (文件調整), style: (程式碼風格調整)。
    • 範例： fix: 修正首頁按鈕失效問題

如何執行 git commit？

指令作用說明
git commit開啟預設編輯器（通常是 Vim），讓你輸入詳細的提交訊息（Commit Message）。
git commit -m "你的提交訊息"最常用！ 直接在指令中輸入提交訊息，無需開啟編輯器。
m表示 message。
git commit -am "訊息"進階/須謹慎！ 跳過 git add 步驟，直接提交**所有已追蹤（Tracked）**但已修改的檔案。
  • 用命令列(CLI)執行。
  1. 確認狀態： 執行 git status，確認所有變更都在 綠色的 Staged 狀態。
  2. 執行提交： 輸入 git commit -m "feat: 建立專案說明文件 readme.md"。
  3. 檢查結果： 再次執行 git status，終端機將會顯示 nothing to commit, working tree clean，表示工作區已經乾淨，與最新的 Git 倉庫版本同步。
  4. 確認commit狀態：使用git log指令確保commit建立完成。
  • 用視覺化工具 (GUI) 執行 (Sourcetree)：
  1. 在 Staged Files 區域確認檔案無誤。
  2. 在右下角的輸入框中，填寫 Commit Message。
  3. 點擊 Commit 按鈕，即可完成提交。
  4. 點擊History按鈕，可以看到整個歷史。點下commit可以看到完整的版本訊息。
