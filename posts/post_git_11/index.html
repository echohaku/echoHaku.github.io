Git 系列分享 - p.9 你的當前位置：HEAD、DETACHED HEAD 與 ORIG_HEAD

November 7, 2025
#Git

在 Git 系列分享 - p.8 Git 的核心力量 - 分支 (Branch) 中，我們學會了使用分支（Branch）來隔離工作，並用「多人交換日記」的比喻來理解 Git 的核心概念。 在那篇比喻中，我們提到了 HEAD 就像是你的「書籤」，它永遠指向你當前正在工作的位置。 今天，我們就要深入探討 HEAD 到底是什麼，以及當這個「書籤」沒有插在「便利貼」（分支）上時，會發生什麼事？這就是所謂的 DETACHED HEAD（分離頭部） 狀態。

一、HEAD：你的「書籤」（當前工作位置）

HEAD 是 Git 中一個非常特殊的指標（Pointer），它只有一個任務：永遠指向你目前所在的位置。 HEAD 的狀態決定了你的下一個 Commit 會被建立在哪裡。

1. 正常的 HEAD（Attached HEAD）

在 99% 的情況下，HEAD 會指向一個分支名稱（例如 main 或 feat/featA）。

  • 運作方式： HEAD → main → Commit_C
  • 比喻： 你的「書籤」（HEAD）插在「藍色便利貼」（main 分支）上，而這張便利貼貼在日記的「第 C 頁」（Commit_C）。
  • 結果： 當你寫下新的一頁（Commit_D）時，你的書籤和藍色便利貼會一起移動到 Commit_D。這是最安全且正常的狀態。

2. HEAD 相關指令

指令作用說明
git status會在第一行顯示 On branch <branch-name>，告訴你 HEAD 目前附著在哪個分支上。
git branch當前 HEAD 所在的分支會以 * 號標記。
git log會用 (HEAD -> <branch-name>) 標示出 HEAD 的位置。
cat .git/HEAD(進階) 直接查看 .git 資料夾中的 HEAD 檔案內容，你會看到 ref: refs/heads/main，證明它指向一個分支。

二、DETACHED HEAD：分離頭部狀態

DETACHED HEAD 不是一個錯誤，而是一種特殊狀態。它發生在 HEAD 不再指向分支名稱，而是直接指向某一個特定的 Commit SHA-1 雜湊值時。

  • 運作方式： HEAD → Commit_C
  • 比喻： 你把「書籤」（HEAD）從「便利貼」（分支）上拿下來，直接插在日記的「第 C 頁」（過去的某一個版本，而非最新版）。你現在的位置和任何便利貼都無關，你「分離」了。

1. 發生情境

DETACHED HEAD 最常在以下情境發生：

  1. 切換到某個特定 Commit： 你想回顧某個舊版本。

    git switch <commit-hash> 
# (例如: git switch 1a2b3c4)
# (舊指令: git checkout 1a2b3c4)

  2. 切換到某個標籤 (Tag)： 你想查看 v1.0 發布時的程式碼。

    git switch v1.0.0

  3. (進階) 互動式 Rebase 過程中： 當 git rebase -i 暫停在某個 Commit 讓你修改時，你也會處於 DETACHED HEAD 狀態。

2. DETACHED HEAD 的「危險」

當你的書籤（HEAD）直接插在「第 C 頁」時：

  • 如果你只是閱讀：沒有任何問題。看完後，git switch main 把書籤插回 main 便利貼即可。
  • 如果你寫了新內容：這才是問題所在。
    • 你在「第 C 頁」後寫了「第 D 頁」（Commit_D）和「第 E 頁」（Commit_E）。
    • 此時，Commit_D 和 Commit_E 不屬於任何分支（沒有任何便利貼貼在上面）。
    • 當你切換回 main 分支時，Git 看不到任何指向 Commit_E 的便利貼，Git 會認為這些 Commit 是沒用的，並在未來某個時間點（垃圾回收機制）將它們永久刪除。

3. 如何儲存 DETACHED HEAD 狀態下的工作？

如果你在 DETACHED HEAD 狀態下意外地做了新的 Commit 並想保留它們，你必須為它們建立一個分支（貼上一張新的便利貼）：

指令作用說明
git switch -c <new-branch-name>(推薦) 立即建立一個新分支，HEAD 會附著到這個新分支上，你的新 Commit 就安全了。
git branch <new-branch-name>效果相同，建立一個新分支指向你當前的 Commit，但 HEAD 仍然分離，你還需要 git switch 一次。

三、ORIG_HEAD：你的「安全備忘錄」

ORIG_HEAD 是 Git 提供的一個臨時備份指標，它像是一個「備用書籤」。

  • 比喻： 在你要進行「撕掉頁面」（reset）或「重新裝訂」（merge）這種複雜操作之前，Git 會自動把 ORIG_HEAD 這個備用書籤，插在你開始操作前的那一頁。
  • 目的： 作為一個安全機制，讓你在操作搞砸時，能有一個快速返回的參考點。

1. 發生情境

ORIG_HEAD 會在執行以下「高風險」指令時被自動建立：

  • git merge
  • git rebase
  • git reset --hard

2. ORIG_HEAD 相關指令

指令作用說明
git reset --hard ORIG_HEAD(超級實用) 撤銷上一次的 merge 或 reset 操作，跳回到執行指令前的狀態。
git diff HEAD ORIG_HEAD比較當前狀態（例如合併後）和執行操作前的狀態有何不同。

小結

HEAD、DETACHED HEAD 和 ORIG_HEAD 是理解 Git 運作方式的關鍵：

  • HEAD： 你的書籤，指向當前分支（便利貼）。
  • DETACHED HEAD： 你的書籤，直接指向某頁 Commit（不是錯誤，是狀態）。
  • ORIG_HEAD： Git 的備用書籤，在你執行高風險操作時自動建立的「還原點」。
