Git 系列分享 - p.13 遠端協作的橋樑：`git remote`、`git pull` 與 `git push`

November 19, 2025
#Git

經過前面幾篇的學習，我們已經掌握了在本地倉庫中管理歷史、分支和 Commit 的各種技巧。然而，Git 的真正強大之處在於其分散式的特性，讓多人可以在不同的地方安全地協作。

今天，我們將探討如何將本地的成果與 GitHub、GitLab 或 Bitbucket 等遠端倉庫（Remote Repository）進行同步。這一切都圍繞著三個核心指令：git remote、git pull 和 git push。

一、git remote

git remote 指令負責管理你的本地 Git 倉庫所認識的遠端 URL 與其「短名」。

1. 什麼是 Remote？

  • Remote（遠端）：一個 Git 倉庫的 URL。
  • Shortname（短名）：代表遠端 URL 的代稱，例如：origin。
  • 預設名稱 origin： 當你使用 git clone 時（這個指令會在下一篇說到，只要想像成是把遠端的東西複製到本地端就可以了），Git 會自動建立 origin 指向原始來源。

2. 常見情境與指令

指令作用說明範例情境
git remote列出所有已配置的遠端短名。想知道你的專案可以同步到哪些地方。
git remote -v列出遠端短名及其對應的 URL（含 Fetch 和 Push 網址）。想確認 origin 實際上指向哪個 GitHub 連結。
git remote add <name> <url>新增一個遠端倉庫。你想將專案同時備份到 GitHub (origin) 和 GitLab (backup)。
git remote remove <name>刪除一個遠端連接。某個遠端倉庫不再使用或網址失效。
git remote rename <old> <new>重新命名遠端短名稱。

二、git pull：從遠端「下載並整合更新」

git pull 的功能是從遠端倉庫抓取（Fetch）最新的變更，並將這些變更與你的本地工作進行整合。

1. 什麼是 Remote？

git pull 實際上是兩個指令的組合：git fetch (抓取) + git merge (合併)。

  • fetch： 僅下載遠端分支上的新 Commit 紀錄和檔案內容，但不修改你的本地工作目錄或本地分支指標。
  • merge： 將下載下來的遠端變更，自動合併到你當前的本地分支。
git fetch + git merge = git pull

2. 發生情境與指令

  • 情境： 在開始工作前，你需要確保你的本地分支（例如 main）擁有隊友最新的 Commit 紀錄。
  • 指令：
    # 從遠端 origin 抓取並合併 main 分支的內容
git pull origin main 

# 設置本地分支追蹤遠端分支後，可簡寫為：
git pull

3. pull 的隱藏策略

git pull 預設是使用 git merge 進行整合。但你也可以強制它使用 rebase 策略：

git pull --rebase
  • 作用： 這樣可以避免在每次 pull 時都產生一個「Merge Commit」，讓你的本地歷史保持一條直線。這在個人工作分支上是個很棒的習慣。

4. Fetch 與 Pull 的實務差異

指令作用是否修改本地工作目錄
git fetch只下載更新，不合併❌ 不會修改
git pull下載並自動合併✔️ 會修改

許多團隊會主張「先 fetch，再自己決定要 merge 還是 rebase」，加強掌控度。

三、git push：將本地成果「發布出去」

當你完成功能或修正 Bug 後，需要用 git push 把本地 commit 傳到遠端。

1. 什麼是git push

git push 會將：

  • 本地 branch 的最新指標位置（HEAD）
  • 該分支的 Commit 歷史 同步到遠端分支。

2. 發生情境與指令

  • 情境： 你完成了新功能並 Commit 到本地分支，現在準備與團隊分享。
  • 指令：
    # 將本地 main 分支推送到遠端的 origin
git push origin main

# 第一次推動新分支時，需要設定追蹤關係
git push -u origin <new-branch> 
# (-u 等同於 --set-upstream，設定後續可簡寫為 git push)

3. Push 被拒絕（Rejected）

常見錯誤訊息： ! [rejected] main -> main (non-fast-forward) 原因是：

  • 遠端已經有其他人推送更新
  • 你的 commit 並不是基於最新的遠端歷史 解法：
  1. 先執行 git pull（將遠端變更拉回來並整合）。
  2. 解決可能發生的衝突（Conflict）。
  3. 再次執行 git push。

四、遠端協作的黃金法則：Pull Before Push

為了確保團隊的歷史線保持同步和一致，請務必遵循以下黃金法則：

  • 在 git push 之前，永遠先執行 git pull。 這能確保你在分享自己的工作成果時，已經將隊友的所有最新變更整合進來，從而將團隊間的衝突和歷史分歧降到最低。

小結

指令目的類型
git remote管理遠端連結（URL & 名稱）設定類
git fetch下載遠端更新，不合併安全操作
git pull下載並合併更新同步本地
git push上傳本地 commit 到遠端發布成果
-u / --set-upstream建立本地分支與遠端的追蹤關係分支管理
