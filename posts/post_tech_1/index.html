Serialize 到底是什麼？

January 19, 2026
#Coding
#Java

名詞定義

序列化 serialization

序列化是指 將記憶體中的物件（Object）依特定序列化機制，轉換成可儲存或可傳輸的線性格式（例如 bytes 或文字格式）過程。
在物件被序列化之後，資料可以被存放在檔案中，或透過網路傳遞給另一個系統，並在必要時再被還原。

注意：序列化是一個廣義概念，實作方式包含 Java 原生序列化、JSON、XML、Protobuf 等，不同機制的行為與限制並不相同。

反序列化 deserializing

反序列化是指將已序列化的資料，依原本的序列化機制，還原為記憶體中的物件狀態。

Java 反序列化的執行順序：

  1. 依 package name 與 class name 尋找 class
    • 若找不到，會直接拋出 ClassNotFoundException
  2. 比對 serialVersionUID
    • 比對序列化資料中記錄的 UID 與目前 class 定義中的 UID
    • 若不一致，會拋出 InvalidClassException

serialVersionUID

serialVersionUID 是 Java 原生序列化中，用來識別 class 版本的 ID，其目的在於控制反序列化時的版本相容性。

為什麼需要 serialVersionUID？

  • 若 class 未明確宣告 serialVersionUID
    • JVM 會根據 class 結構（欄位、方法、繼承關係等）自動產生
    • 只要 class 結構稍有變動，UID 就可能改變
    • 導致舊資料在反序列化時失敗

範例說明

假設：

  • User class 原本包含欄位 (name, age, phone)
  • 將其序列化並儲存在檔案 A 中
  • 後續版本中刪除了 phone 欄位，並修改了 toString() 方法 若 未明確指定 serialVersionUID：
  • JVM 可能重新計算 UID
  • 嘗試反序列化檔案 A 時，會拋出 InvalidClassException 若 有明確指定 serialVersionUID：
  • JVM 會認定這仍是同一個 class
  • 即使欄位有所差異，仍允許反序列化（新欄位可能為預設值）

serialVersionUID 通常如何設定？

  • 直接指定固定值 private static final long serialVersionUID = 1L;
  • 由工具或 JVM 根據 class 結構產生出一個 64-bit hash value（通常唯一）
    • package name
    • class name
    • 繼承關係
    • 非 private 的方法與欄位

所有 class 都設定為 1L的時候，JVM會允許反序列化，但可能會造成

  • 新欄位可能會直接是預設值
  • 舊資料結構可能與新邏輯不相容
  • 問題不一定會在反序列化當下立即暴露

什麼時候會建議都設定為 1L ？

  • DTO
  • Cache Object
  • Session Object
  • 欄位變動後仍可以向後相容

什麼時候應該要調整SerialVersionUID？

  • 欄位意義改變
  • 移除欄位
  • 改變繼承結構
  • 改變 transient / non-transient
public class ExampleClass implements Serializable { 
	private static final long serialVersionUID = 1L;
	// class details... 
}

在JAVA當中什麼時候該 implements Serializable？

判斷標準：「物件會被轉換成 byte stream 並在 JVM 之外存活或傳遞」時，才需要 implements Serializable

  • 需要
    • session / cache / message queue
    • 要寫成檔案的的class
  • 不需要的情境
    • REST API request / response（JSON / XML）
    • Controller / Service / Repository
    • 單純在 JVM 內部傳遞的物件

層級應用層 / 資料交換層JVM / 語言層
使用目的系統間資料交換JVM 物件狀態保存與傳遞
常見使用場景REST API、Frontend ↔ BackendSession、Cache、RMI、File
輸出格式JSON（文字）Binary byte stream
是否跨語言✅ 是❌ 否（Java only）

版本相容機制API contract / schemaserialVersionUID
實作方式Reflection / Getter / AnnotationObjectOutputStream
可讀性高（人可讀）低（binary）
適合長期儲存⚠️ 視情況⚠️ 不建議（版本風險）
效能中等較快（但耦合高）

常見誤解釐清1: API Response 要 implements Serializable

錯誤原因：混淆「JSON 序列化」與「Java 原生序列化」 REST API 回傳 JSON 時：

  • Jackson 將 Object 轉為 JSON
  • 完全不使用 Serializable

常見誤解釐清2: Entity / DTO 都加 Serializable 比較安全

錯誤原因：把 Serializable 當成通用標記介面

  • Entity：依使用情境決定
  • DTO（API）：不需要
  • Session / Cache Model：需要
