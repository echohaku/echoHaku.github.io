Git 系列分享 - p.12 歷史控制：`git reset`、`git revert` 與 `git cherry-pick`

November 18, 2025
#Git

在Git 系列分享 - p.11 歷史重寫與整理：git rebase 和 squash 的魔法中，我們學會了如何使用 git rebase 來清理本地的 Commit 歷史。今天，我們將探討另外三個控制 Git 歷史的關鍵指令，它們分別代表了三種處理 Commit 的態度：破壞性撤銷、安全撤銷，以及精準複製。

一、git reset：破壞性撤銷（時光倒流並撕掉頁面）

git reset 是一個強大的「時光機」指令，它能將你的 HEAD（書籤）和當前分支指標（便利貼）移回到歷史中的某個 Commit。

1. 定義與比喻

  • 定義： git reset 會移動 HEAD 和分支指標，並根據不同的模式（--soft, --mixed, --hard），來處理被「撤銷」 Commit 中的內容。它是一種破壞性操作，因為它會讓後面的 Commit 看起來像是從來沒發生過。
  • 書籤比喻： 想像你寫日記時發現寫錯了，你將書籤（HEAD）退回到前幾頁，並且可能將後面的頁面（Commit）全部撕掉或塗改。

2. 三種模式（處理內容的方式）

模式作用於 Commit作用於暫存區作用於工作目錄比喻（被撕掉頁面上的內容）
--soft退回保留保留內容已用原子筆寫完，但你讓書籤退回，告訴自己「這一頁不算數」，內容留在桌上。
--mixed退回清除保留內容從原子筆稿退回成鉛筆草稿（清除暫存區），內容留在桌上。
--hard退回清除清除直接撕掉頁面，丟進垃圾桶（清除工作目錄和暫存區）。最危險，請謹慎使用！

3. 發生情境與指令

  • 情境： 發現最新的 Commit 寫錯了，且該 Commit 尚未推送到遠端（未共享）。
  • 指令：
    # 常用：撤銷最近一次 Commit，保留內容在工作目錄中繼續修改
git reset HEAD~1 

# 危險：完全拋棄最近三次 Commit 的所有內容
git reset --hard HEAD~3 

# 撤銷所有暫存區內容，回到工作目錄（清除鉛筆稿，保留內容在紙上）
git reset HEAD <file>

二、git revert：安全撤銷（承認錯誤並寫一篇新的日記）

git revert 是一種非破壞性的撤銷方式。它不會移動 HEAD 或刪除任何 Commit，而是透過新增一個 Commit 來抵銷歷史中的某個變更。

1. 定義與比喻

  • 定義： git revert 會創建一個全新的 Commit，這個新 Commit 的內容是目標 Commit 的反向變更。
  • 書籤比喻： 你的日記已經寫到第 10 頁，發現第 5 頁寫錯了。你不會撕掉第 5 頁，而是在第 11 頁寫一篇新的日記，內容是：「我發現第 5 頁寫錯了，現在我把它的內容修正回來。」
  • 優勢： 歷史線是連續的，Git 保留了「錯誤發生」和「錯誤被修正」的完整軌跡。

2. 發生情境與指令

  • 情境： 發現已經推送到遠端（已共享）或已合併到主線的 Commit 有 Bug，需要安全地取消該變更。
  • 指令：
    # 撤銷特定 Commit 的變更，並產生一個新的 Revert Commit
git revert <commit-hash> 

# 撤銷最新的 Commit，但不自動Commit，讓你手動修改後再提交
git revert HEAD --no-commit
  • 鐵則： 所有已共享、已推送的 Commit 錯誤，一律使用 git revert 處理。

三、git cherry-pick：精準複製（挑選最棒的一段內容）

git cherry-pick 允許你將歷史中任何一個單獨的 Commit，複製到當前 HEAD 所在的分支上，並產生一個新的 Commit。

1. 定義與比喻

  • 定義： 複製單一 Commit 的程式碼變更，並將其重新應用（Apply）到當前分支，形成一個新的 Commit ID。
  • 書籤比喻： 你正在寫你的藍色便利貼（feature 分支）。你看到你的朋友小明在另一條紅色的便利貼上寫了一段超級棒的內容（一個 Commit），你決定把那段內容單獨抄寫一份，貼在你的藍色便利貼最新位置。

2. 發生情境與指令

  • 情境 A (Hotfix)： 你在 develop 分支上緊急修了一個 Bug，但 main 分支也需要這個修正。你不需要合併整個 develop，只需要挑選這個 Bug Fix 的 Commit 到 main。
  • 情境 B (重複利用)： 某個 Commit 內容非常有價值，但它位於一個後來被廢棄的分支上，你可以將其挑選出來。
  • 指令：
    # 將目標 Commit (hash) 複製到當前分支，並產生新的 Commit ID
git cherry-pick <commit-hash> 

# 複製 Commit 但不自動提交，讓你可以在提交前修改或調整訊息
git cherry-pick <commit-hash> -n

四、值得分享的知識點：處理失誤的決策樹

當你發現一個錯誤或需要移動變更時，請依循以下原則來選擇指令：

  1. 錯誤發生在哪裡？
    • 在本地且未推送？ → git reset (可以安全地重寫歷史)
    • 已推送到遠端且已共享？ → git revert (非破壞性，保留軌跡)
    • 沒有錯誤，只是想確保我的修改在最後面 → git rebase
  2. 你是想整合還是想複製？
    • 想將一整個功能（多個 Commit）的變更帶入？ → git merge 或 git rebase
    • 只想將單一、獨立的 Commit 帶入？ → git cherry-pick
指令目的破壞性/安全性適用性
git reset撤銷本地歷史，回到過去某個 Commit。破壞性 (重寫歷史)本地私人分支
git revert創建反向 Commit，安全地取消變更。非破壞性 (新增歷史)遠端共享分支
git cherry-pick複製單一 Commit 的變更到當前位置。非破壞性 (新增歷史)跨分支複製

小結

reset、revert 和 cherry-pick 都是 Git 歷史管理的瑞士刀。請務必將 reset 留給你的私人工作區，而將 revert 留給團隊共享的分支。一旦掌握了這三者，你就能徹底控制 Git 專案中的每一次變動。 下一篇：Git 系列分享 - p.13 遠端協作的橋樑：git remote、git pull 與 git push
