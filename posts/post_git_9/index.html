Git 系列分享 - p.7 版本控制的篩選器 - .gitignore 的使用與規則

October 23, 2025

在Git 系列分享 - p.6 正式提交一個commit囉中，我們學會了如何使用 git add 和 git commit 為專案建立乾淨且有意義的歷史紀錄。

然而，在你的專案資料夾中，總是會存在一些你不希望被 Git 納入追蹤、也不想提交到版本庫的檔案。這些檔案會不斷被 git status 標記為 Untracked files，造成混亂。這時，我們就需要版本控制的篩選器：.gitignore。

一、.gitignore：使用情境與目的

.gitignore 是一個簡單的純文字檔案，它告訴 Git 應該忽略哪些檔案和資料夾。它的目的在於：

目的說明檔案範例
保護隱私環境變數檔通常包含 API Key、密碼等敏感資訊，必須阻止其上傳。.env, config.local
避免冗餘編譯產生的檔案或大型函式庫可以隨時重新生成，不應浪費空間納入版本庫。*.exe, *.log, node_modules/
保持乾淨忽略個人編輯器或作業系統產生的暫存檔，讓團隊成員之間不受彼此環境的干擾。.vscode/, .DS_Store

二、如何建立與添加規則？

.gitignore 檔案必須建立在你的專案根目錄下，這樣 Git 才能在專案的任何地方套用這些規則。建立後，記得將 .gitignore 本身加入 Git 追蹤，以便團隊成員共享規則。

1. 命令列 (CLI) 執行

在專案根目錄下，你可以使用指令直接建立這個檔案並寫入內容：

# 建立一個名為 .gitignore 的檔案
touch .gitignore

# 編輯檔案，例如加入規則：
echo "node_modules/" >> .gitignore
echo "*.log" >> .gitignore

或是直接建立完檔案後，將.gitignore 檔案加入 Git 即可。

2. 視覺化工具 (GUI) 執行 (以 Sourcetree 為例)

雖然 Sourcetree 等工具不一定有內建的 .gitignore 編輯器，但你可以透過以下方式處理：

  1. 手動建立： 透過檔案總管或編輯器，在專案根目錄手動建立 .gitignore 檔案。
  2. 忽略未追蹤檔案： 在 Sourcetree 的 File Status 介面中，右鍵點擊任何 Untracked 的檔案（例如 .env），通常會有 “Ignore” 選項，你可以選擇「永遠忽略這個檔案」或「永遠忽略這個類型」，Sourcetree 會自動將規則寫入你的 .gitignore 檔案中。

三、.gitignore 的撰寫規則（語法）

在 .gitignore 檔案中，每行代表一個忽略規則。

規則語法作用說明範例
檔案名稱忽略專案內所有該名稱的檔案。secret.txt
萬用字元 (*)匹配零個或多個字符。*.log (忽略所有 .log 結尾的檔案)
目錄結尾 (/)匹配整個資料夾，包括內部所有內容。temp/
例外規則 (!)排除之前已被忽略的檔案或目錄。!temp/keep.txt (忽略 temp 資料夾，但排除 keep.txt)
路徑開頭 (/)僅匹配專案根目錄下的該檔案或資料夾。/build (只忽略根目錄下的 build 資料夾)

四、.gitignore 的重要注意事項（關鍵）

這是在使用 .gitignore 時最常遇到的陷阱，請務必牢記：

1. .gitignore 僅對「未追蹤 (Untracked)」的檔案有效

  • 原理： Git 會忽略那些從未被 git add 過的檔案。
  • 狀況： 如果一個檔案在你的 Commit History 中已經存在（即它已經被 Git 追蹤過），即使你將它加入 .gitignore，Git 仍會持續追蹤它，並且當它被修改時，git status 還是會顯示 Modified。

2. 如何移除已被追蹤(Tracked)的檔案？

如果你不小心提交了敏感或多餘的檔案（例如 config.env），你需要執行以下操作，將它從 Git 的追蹤名單中移除，但保留本地檔案： 這樣的方式僅限於，不改變過去的commit，但確保未來將不再追蹤該檔案。

# 1. 將檔案從 Git 倉庫中移除，但保留電腦上的實體檔案。
# git rm --cached 會在暫存區建立一個「刪除」該檔案的動作快照。
# 同時會檢查repository和working directory檔案內容是否相同，
# 都要相同才會去做執行！避免誤刪未儲存的變動。
# 執行 git status 會看到"deleted: config.env"
# 補充： --cached會讓檔案留在本機上，git rm config.env則是直接把檔案刪掉！
git rm --cached config.env

# 2. 將該檔案加入 .gitignore，確保它不會再被追蹤
echo "config.env" >> .gitignore

# 3. 提交這個「移除追蹤」和「忽略規則」的變更
git commit -m "chore: 將 config.env 移出 Git 追蹤"

3. 從歷史紀錄中永久移除機敏資訊 (歷史重寫)？

如果機敏資訊（例如密碼或 API Key）已經被提交（Commit），則該檔案的快照已經永久嵌入到所有相關的歷史 Commit 中。若想要改變過去的 Commit 歷史，必須要重寫歷史。

  • 方式： 官方建議使用 git filter-repo 或 BFG Repo-Cleaner 等工具來重寫歷史，這會將指定的檔案從所有 Commit 快照中強制移除。
  • 指令範例 (使用 git filter-repo 移除 config.env)：
    • 註：執行此指令前，請確保已安裝 git filter-repo。
    git filter-repo --forget-blob --path config.env
  • 重寫歷史的後果：
    • 所有 Commit 中的機敏檔案都會被移除。
    • Git 歷史會被「重寫」。
    • 舊的 Commit SHA 雜湊值（版本編號）全部都會改變。
  • 重要警告（遠端推送）：
    • 重寫歷史是一項破壞性操作。
    • 如果未來你需要將這個被重寫過歷史的倉庫推送到遠端，你必須使用 git push --force 強制覆蓋遠端歷史。這應是最後手段，且必須在通知所有團隊成員後才能進行，因為會影響所有人的本地版本庫。

4. 別忘了提交 .gitignore 本身！

.gitignore 檔案本身是一個重要的專案配置，它必須被 git add 和 git commit，這樣才能讓你的所有團隊成員共享相同的忽略規則，確保大家的工作目錄都能保持乾淨。

小結

.gitignore 是維護專案清潔與安全的關鍵工具。通過學習它的使用，你就能確保版本庫中只包含真正需要的原始碼。
