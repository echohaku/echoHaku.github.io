Git 系列分享 - p.17 進階工作流：Hooks、Patch 與 Cherry-Pick

November 27, 2025
#Git

我們已經學習了標準的 Git 指令、歷史管理和團隊協作模型（如 Git Flow）。然而，有時我們需要更精確地控制程式碼的傳輸、強制執行團隊規範，甚至處理離線的程式碼交換。

本篇將介紹三個屬於進階「配置與工作流」層次的工具：Git hooks（自動化規範）、Git patch（離線交換） 以及 git cherry-pick（精準移植）。

一、Git hooks：專案的自動化守衛

Git hooks 不是一個 Git 指令，而是一系列預定義的腳本（Scripts）。當 Git 倉庫中發生特定事件（如 Commit 前、Push 前、接收 Commit 後）時，Git 會自動執行這些腳本。

1. 定義

  • 定義： 一種客製化的自動化機制，讓開發者能夠在 Git 流程的特定階段（Hooks）插入自定義的檢查或動作，以避免不必要的錯誤並及時阻止進入下一個階段。這樣可以增加生產力，節省除錯及事後緊急bug fix的時間，CICD前也可以先檢查！
  • 檔案位置： 所有的腳本都存放在每個本地倉庫的 .git/hooks/ 目錄下。

2. 發生情境與腳本類型

Hooks 類型觸發時機典型應用情境
pre-commit在你輸入 Commit 訊息之前執行 Lint（程式碼風格檢查）、自動格式化程式碼、執行單元測試。
commit-msg在你輸入 Commit 訊息之後檢查 Commit 訊息格式（例如：必須以 feat: 或 fix: 開頭）。
pre-push在程式碼推送到遠端之前確保分支已通過所有測試、防止你將敏感分支推送到公開倉庫。
post-receive遠端倉庫接收到 Push 之後在伺服器上自動觸發網站部署（CI/CD）。

3. 相關指令與知識點

  • 配置： Hooks 腳本通常使用 Shell、Python 或 Node.js 編寫。
  • 本地限制： 預設的 Hooks 是本地且不會被 Commit 到 Git 倉庫的。團隊通常需要使用 Husky 等工具來強制或標準化 Hooks 配置。
  • 中斷流程： 如果一個 Hooks 腳本以非零狀態碼退出，它將中斷當前的 Git 流程（例如：pre-commit 腳本失敗會阻止 Commit 發生）。

二、Git patch：離線程式碼交換

Git patch（補丁）是一種將一個或多個 Commit 的變更內容，打包成一個標準文字檔（.patch 或 .diff 檔）的方法。它主要用於無法直接使用 Git 遠端傳輸的環境。

1. 定義與比喻

  • 定義： 一個包含程式碼變更、Commit 訊息、作者和時間戳等資訊的純文字檔。這個檔案可以透過電子郵件、USB 隨身碟等任何方式傳輸。

2. 發生情境與指令

  • 情境 A (貢獻給 Email 驅動的專案)： Linux 核心等大型開源專案，要求貢獻者通過郵件清單提交變更。
  • 情境 B (離線或受限網路)： 開發者處於隔離或防火牆環境，無法直接 push 或 pull。
  • 相關指令：
指令作用說明
發送變更 (format-patch)將一個或多個 Commit 轉換為格式化的 .patch 檔案。
接收變更 (git am)讀取 Patch 檔案，並將其作為一個完整的 Commit 應用到當前分支。
應用變更 (git apply)不包含 Commit 資訊，僅將 Patch 檔中的程式碼變更應用到工作目錄。

三、git cherry-pick：精準的 Commit 移植術

git cherry-pick 是一個功能強大的指令，它能讓你將單一或選定的 Commit 從歷史中的某個地方，複製並應用到當前分支上。

1. 定義

  • 定義： 複製一個現有的 Commit，並將其內容和 Commit 訊息應用為一個全新的 Commit 到另一個分支的頂端。
  • 關鍵差異： 因為是「複製」而不是「移動」，所以會生成一個新的 Commit SHA。

2. 發生情境與指令

  • 情境 A (Hotfix 移植)： 你在 release 分支上修復了一個小 Bug，但這個 Bug 的修復也需要應用到還在開發中的 feature/big-refactor 分支上。
  • 情境 B (Backporting)： 將主線（main）上的某個特定功能或優化，移植到舊版發佈分支（release-v1.0）上。
  • 指令：
    # 將特定 Commit ID 的變更，複製到當前分支
git cherry-pick <commit-hash>

# 選取多個連續 Commit
git cherry-pick hashA^..hashB 

# 僅應用程式碼變更，但不自動生成 Commit
git cherry-pick -n <commit-hash> 
# (類似 git apply，但保留了 Commit 訊息)

四、總結：進階工具的應用場景

工具核心目的應用時機屬於什麼層次？
Git hooks自動化執行程式碼檢查與流程規範。Commit/Push 前：強制風格檢查。專案配置與團隊規範
Git patch離線傳輸單一或多個 Commit 變更。網路受限或郵件協作：將變更作為附件發送。非標準協作工作流
git cherry-pick精準移植單一 Commit 到非起源的分支。Hotfix 雙向移植：將修復從一個分支帶到另一個分支。精確流程控制與維護
